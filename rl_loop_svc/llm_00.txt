You are an expert machine learning systems engineer tasked with implementing a production-ready Reinforcement Learning microservice for a university research project.
The goal of this microservice is to train a Prompt Rewriter model using Proximal Policy Optimization (PPO) so that rewritten prompts improve ICD-10 code generation accuracy from a downstream medical coding LLM.
This microservice is part of a larger pipeline involving clinical notes, prompt rewriting, ICD coding generation, evaluation, and reward computation.
Your task is to implement the Reinforcement Learning block that receives rewards and updates the Prompt Rewriter model using PPO.

# SYSTEM CONTEXT


Pipeline Overview:

1. User provides a clinical note.
2. The Prompt Rewriter model rewrites the prompt.
3. The rewritten prompt is sent to an ICD-10 coding LLM.
4. The coding LLM outputs predicted ICD codes.
5. An evaluation service compares predicted codes with ground truth.
6. A Reward Metrics service computes a reward score.
7. The reward and generation metadata are sent to the Reinforcement Learning microservice.
8. The RL microservice updates the Prompt Rewriter policy using PPO.

The Reinforcement Learning microservice DOES NOT run the ICD coding LLM.

It only receives: (why original and rewritten prompt??)

- original prompt
- rewritten prompt
- reward score
- log probabilities from the policy
- optional value estimates

and performs PPO updates on the Prompt Rewriter model.


# GOAL OF THE RL MICROSERVICE

Train a Prompt Rewriter policy that generates rewritten prompts that maximize downstream ICD coding accuracy.
The optimization objective is based on Proximal Policy Optimization (PPO).
The policy being optimized is a small language model (SLM) such as Phi-3-mini.

The RL microservice must:
1. Store rollouts
2. Compute advantages
3. Perform PPO updates
4. Update model weights
5. Serve updated policy checkpoints

---------------------------

---------------------------

The microservice receives training samples in the following format:

  "rewritten_prompt": "prompt generated by the prompt rewriter",
  "reward": float,
  "log_prob": float,

Optional fields:

  "metadata": {...}

---------------------------
RL STATE DEFINITION
---------------------------

State (s):

The state corresponds to the original clinical note or parsed prompt context.

s = original_prompt

Action (a):

The action is the rewritten prompt generated by the Prompt Rewriter model.

a = rewritten_prompt

Reward (r):

Reward is computed externally by the Reward Metrics block.

Examples:

1. Exact ICD match → reward = 1.0
2. Partial match → reward = 0.5
3. Incorrect prediction → reward = 0.0

The RL service treats reward as a scalar value.

---------------------------
MODEL ARCHITECTURE
---------------------------

Policy Model:

Prompt Rewriter language model.

Example model:

Phi-3-mini

The policy outputs token sequences representing rewritten prompts.

The policy must support:

- sampling
- log probability computation
- gradient updates

Value Model:

The value function estimates V(s) for PPO.

This may be implemented as:

1. a separate value head
2. shared transformer backbone

---------------------------
ROLLOUT STORAGE
---------------------------

The microservice must store rollouts for PPO updates.

Each rollout must contain:

- state
- action
- reward
- log_prob_old
- value_estimate

Example:

rollout = {
    state,
    action,
    reward,
    log_prob_old,
    value
}

Rollouts should be stored in batches.

Batch size should be configurable.

---------------------------
ADVANTAGE COMPUTATION
---------------------------

Advantages should be computed using Generalized Advantage Estimation (GAE).

Formulas:

delta_t = r_t + gamma * V(s_{t+1}) - V(s_t)

A_t = delta_t + gamma * lambda * A_{t+1}

Hyperparameters:

gamma = 0.99
lambda = 0.95

---------------------------
PPO OBJECTIVE
---------------------------

The PPO clipped objective must be implemented.

ratio = exp(log_prob_new - log_prob_old)

objective:

L = min(
    ratio * advantage,
    clip(ratio, 1 - epsilon, 1 + epsilon) * advantage
)

Recommended hyperparameters:

epsilon = 0.2

Loss components:

policy_loss
value_loss
entropy_bonus

Total loss:

loss = policy_loss + c1 * value_loss - c2 * entropy

Typical coefficients:

c1 = 0.5
c2 = 0.01

---------------------------
TRAINING LOOP
---------------------------

The RL microservice must perform the following steps:

1. Collect rollout samples
2. Store them in a rollout buffer
3. Compute advantages and returns
4. Run multiple PPO epochs
5. Update policy parameters
6. Save checkpoints

Training must be configurable.

Example parameters:

rollout_batch_size = 128
ppo_epochs = 4
mini_batch_size = 32
